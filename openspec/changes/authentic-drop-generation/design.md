# Design — authentic drop generation

## Context

Drops currently come from hand-authored weighted tables (`DROP_TABLES` in `content.ts`) over ~15 invented `GEAR` templates, with three tiers selected by `DifficultyDef.dropTier`. Meanwhile the authentic Blue Burst item dataset is already extracted (`src/engine/data/item-table.json` via `extract-item-table.mjs`), `Character.sectionId` exists (immutable, chosen at creation, currently inert), and the newserv clone provides the canonical drop data:

- `system/tables/common-table-v3-v4.json` — non-rare generation tables, keyed `Episode:Mode:Difficulty:SectionID` with inheritance (previous section ID → previous difficulty → Normal mode). Field semantics: `CommonItemSet::Table::RootT` comments in `src/CommonItemSet.hh`; logic: `src/ItemCreator.cc`.
- `system/tables/rare-table-v4.json` — rare specs keyed `{Mode:{Episode:{Difficulty:{SectionID:{EnemyType | Box-<Area>: [[prob, item], ...]}}}}}`, probabilities out of 2^32 or as fraction strings.

Constraints: the engine is pure and deterministic (all randomness via `engine/rng.ts`; replays of `(runId, seed)` must reproduce identical logs and loot), saves are versioned, and combat math is not touched by this change.

## Goals / Non-Goals

**Goals:**

- Enemy and box drops generated by a faithful port of newserv's common-item pipeline, keyed by the character's section ID, the run difficulty, and the area.
- Rare gear drops from the sliced rare table, rolled at the authentic point in the pipeline.
- Items carry the generated variance (grind, attribute bonuses, special, slots, unit star variance) — stored and displayed, inert in combat.
- Checked-in, reproducible datasets; no inheritance resolution or table math surprises at runtime.
- Deterministic replays preserved.

**Non-Goals:**

- Wiring % bonuses, specials, or slot contents into combat resolution (follow-up change).
- Technique system / tech disks, mags, rare tools, kill-count ("unsealable") items, Episodes 2/4, Battle/Challenge modes, Very Hard difficulty.
- Loot filter redesign (star→rarity mapping keeps the existing filter semantics).

## Decisions

### D1: Resolve table inheritance at extraction time, not runtime

`extract-common-table.mjs` (new script, same pattern as the four existing extractors) walks the inheritance chain (previous section ID → previous difficulty → Normal mode) and emits fully-resolved tables for the slice we use: **Episode 1 × Normal game mode × {Normal, Hard, Ultimate} × all 10 section IDs**. The engine reads a plain `resolved[difficulty][sectionId]` record.

*Why:* the inheritance rules are a serialization detail of newserv's file format; porting them into the engine adds failure modes for zero benefit. Baking also lets the script validate the slice (all offsets present, table shapes correct) at generation time. Alternative considered — runtime resolution — rejected as pure complexity.

Tables carried (per `RootT`): weapon type/subtype-base/subtype-area-length, grind, armor type + bias + slot counts, enemy rt-index meseta ranges / drop probs / item classes, box meseta ranges, bonus value/spec/type tables, special mult/percent, unit max stars, tool class table, box item-class table. Technique tables are **not** carried (see D6).

### D2: Distribution-faithful port, not bit-identical RNG

`drop-gen.ts` reproduces `ItemCreator`'s table lookups, index-probability sampling, and formulas (integer truncation included), but draws randomness from the run's seeded `Rng` rather than PSO's crypt stream. Same distributions, our determinism.

*Why:* replays only need self-consistency, which the seeded RNG already guarantees. Bit-exact PSO RNG would buy nothing observable and cost a second RNG implementation.

### D3: Drop pipeline shape and rare-check placement

Per enemy kill: `dropProb[rt_index]` roll (nothing on failure) → rare-spec roll for this enemy type (each spec independently at `prob / 2^32`; first hit wins) → otherwise `itemClass[rt_index]` → class-specific generation. Per box: rare-spec roll for `Box-<Area>` → `boxItemClass[class][area_norm]` (which includes meseta and empty outcomes). This matches `on_monster_item_drop` / `on_box_item_drop` in `ItemCreator.cc`.

Rare scope: specs whose item code resolves to a weapon/frame/barrier/unit in `item-table.json` are kept at extraction time; rare tool/mag/meseta specs are dropped from the dataset. Rare items are minted from their item-code definition with rare bonus generation (spec-5 bonus rolls, random area column) and rare grind; they are always `rarity: "rare"`.

### D4: `rt_index` and rare-table keys resolved at extraction time via `statsType`

Enemies' existing `statsType` strings (from the enemy-stats extraction) are mapped to newserv `EnemyType` names in the extractor, which emits per-enemy `rtIndex` and validates that **every wired enemy** resolves to an rt-index row and that every rare-table `Where` key we keep matches a wired enemy or box area. Mismatches fail the extraction loudly.

*Why:* the fiddly name-alignment problem is pushed to generation time where it's a build error, not a silent runtime miss.

### D5: Additive item-model fields + star→rarity mapping

`items.ts` gains optional fields: weapons `{ bonuses: {native?, aBeast?, machine?, dark?, hit?}, special? }`, frames/barriers `{ slots? }` (count only; slot contents are future work), units keep star variance implicitly via stats. All generated items carry `stars` (from `item-table.json`) and their PSO item `code`.

Rarity label mapping: rare-table drops → `rare`; otherwise the existing star bucket from the item-parameter-data loader (0–3 common, 4–8 uncommon, ≥9 rare). The existing loot filter (`autoSellBelow` + `alwaysKeep: ["rare"]`) keeps working unmodified.

### D6: Tools — usable four unchanged, rest inert, tech disks excluded at extraction

Tool-class drops resolving to mates or Moon Atomizers route to the existing consumable supply; grinder-class drops route to the existing fungible grinder count. Every other tool class mints an inert `Tool` item (new `ItemKind`) carrying code, name, and sell value from the tool table — sellable, filterable by value, not usable. Tech-disk classes get their weights zeroed in the extracted tool-class table so they can never be selected (no runtime re-roll special case). The `item-table.ts` loader exposes tool definitions (already extracted, currently hidden).

*Why zero-at-extraction over runtime re-roll:* keeps `drop-gen.ts` free of policy branches and keeps the distribution well-defined.

### D7: Meseta goes drop-based

The per-kill stat-row meseta award in `run.ts` is removed. Enemy meseta drops via `enemyMesetaRanges[rt_index]` (gated by the drop-anything probability and the meseta item class); box meseta via `boxMesetaRanges[area_norm]`. `DifficultyDef.mesetaMult` still applies to dropped meseta as the economy knob; `dropTier` is retired.

### D8: `area_norm` mapping on `AreaDef`

Areas gain a derived `area_norm` (Forest 1 = 0, …). Boss floors use the **next** section's first area (Dragon → Caves 1); the final boss uses the last non-boss area. Encoded as a small explicit map next to `AREAS` rather than inferred from floor numbers.

### D9: Save version 4, migrate in place

Existing saved items lack the new optional fields — migration stamps nothing (fields are optional) but the version bump records the shape change; inventories with legacy `GEAR` template items survive as-is (their `id`s and stats are self-contained). Shop stocks referencing removed `GEAR` templates are regenerated on load.

## Risks / Trade-offs

- **Economy shock** (kills often pay nothing; commons flood in) → explicit retune task: `autoSellBelow` default, shop prices, `mesetaMult`; distribution sanity test asserts average meseta/run stays within a tuned band.
- **Formula drift vs. `ItemCreator.cc`** (integer truncation, index-probability edge cases, off-by-one in area indexing) → spot-check unit tests with hand-verified expected values from the newserv source for known table cells; property test that generated weapons always resolve to valid item codes.
- **`statsType` ↔ `EnemyType` name mismatches** → extraction-time validation (D4) fails the build, not the run.
- **Dataset bloat** (10 section IDs × 3 difficulties) → resolved slice carries only the tables listed in D1; expected well under the size of `item-table.json`.
- **Legacy shop stock references break** → regenerate stocks on v4 load (D9); shop draws from the authentic dataset already, only placeholder-template references need purging.

## Migration Plan

1. Land extraction scripts + datasets (inert — nothing reads them yet).
2. Land `drop-gen.ts` with tests against the datasets.
3. Switch `run.ts` call sites, remove `DROP_TABLES`/`GEAR`/stat-row meseta, bump `SAVE_VERSION` to 4 in the same commit (the engine change and save change are atomic).
4. Retune economy; update UI item rendering.

Rollback: revert the switch commit; datasets and generator are additive and harmless on their own.

## Open Questions

- Armor/shield DFP/EVP variance uses the `dfpRange`/`evpRange` fields already in `item-table.json` (uniform roll within range, per `generate_common_armor_or_shield_type_and_variances`) — confirm during implementation that the extracted ranges match newserv's interpretation.
- Whether Al Rappy / Nar Lily (rare enemy variants the stage generator can roll) use their own rt-index rows and rare specs — resolve during extraction validation.
